⚙️ System Design Architecture & Scalability Narrative ▼

Sliding Window Counter Details

Enforces strict rolling window thresholds by executing ZREMRANGEBYSCORE and ZCARD inside an atomic Redis Lua script. Evicts expired elements in score ranges and checks cardinality.

Token Bucket Details

Enables bursts using dynamic capacity hashes containing last updated timestamp and token balances. Refills mathematically based on time elapsed: refilled = tokens + (elapsed_ms * rate).

Operational Scale Capacity

Memory Footprint: Sliding Window: ~2.4MB for 10k users. Token Bucket: ~0.9MB for 10k users.
Scaling Plan: 3 Express backend nodes behind an Nginx gateway scale linearly up to 50k req/sec. For greater capacity, shard using consistent hashing keys.

Defensive Edge Case Handling

1. Duplicate Timestamps
Using Redis ZSET, members must be unique. We avoid timestamp overrides by appending random request UUIDs to scores, allowing multiple concurrent checks.
2. Clock Skew Resilience
If a client sends future timestamp queries to bypass sliding logs, the API detects clock drift (>5s ahead) and rejects it with 400 Bad Request.
3. Horizontal Race Conditions
Single-threaded Redis Lua scripts execute check-then-count pipelines atomically inside Redis, preventing double allowances.

Check Quota Limit

Post a user request to verify sliding window allowances. Can authenticate using user ID in payload or API Key header.

METHOD & URL:
POST https://rate-limiter-three.vercel.app/api/v1/check-limit
HEADERS:
Content-Type: application/json
X-API-Key: rate_key_7480c171977... (Alternative)
X-User-Tier: pro (Alternative if no key)
X-Rate-Limiter-Algo: sliding | bucket (Enforces algorithm selection)
